# Протокол сохранения контекста между сессиями

## Когда сохранять чекпоинт
- Диалог идёт долго (>15-20 сообщений с кодом)
- Система начала сжимать предыдущие сообщения
- Задача ещё не завершена и есть незаконченные шаги
- **НЕ ЖДАТЬ** пока контекст потеряется — сохранять ЗАРАНЕЕ

## Формат чекпоинта (файл: memory/checkpoint.md)

```markdown
# Чекпоинт сессии
Дата: YYYY-MM-DD HH:MM

## Текущая задача
[Что делаем — краткое описание]

## Что сделано
- [x] Шаг 1 — описание
- [x] Шаг 2 — описание

## Что осталось
- [ ] Шаг 3 — описание
- [ ] Шаг 4 — описание

## Ключевые решения
- Решение 1: почему так
- Решение 2: почему так

## Изменённые файлы
- `path/to/file1.js` — что изменено
- `path/to/file2.js` — что изменено

## Контекст для продолжения
[Всё что нужно знать чтобы продолжить работу без перечитывания всего диалога]

## Этап по CLAUDE.md workflow
Текущий этап: [architect / код / tester / code-reviewer / security-reviewer / docs / DoD]
Следующий шаг: [что делать дальше]
```

## Процедура для пользователя
1. Я пишу: "Контекст на исходе, сохранил чекпоинт"
2. Пользователь создаёт новую сессию
3. Пользователь пишет: "продолжи" (или любое сообщение)
4. Я читаю memory/checkpoint.md и продолжаю с того места

## Процедура для меня (Claude)
### При начале новой сессии:
1. Прочитать MEMORY.md (загружается автоматически)
2. Проверить есть ли memory/checkpoint.md
3. Если есть — сообщить пользователю что есть незаконченная работа

### При длинном диалоге:
1. Отслеживать объём работы
2. Когда чувствую что контекст на исходе — сохранить чекпоинт
3. Уведомить пользователя
4. НЕ пытаться продолжать работу с деградированным контекстом

### После завершения задачи:
1. Удалить memory/checkpoint.md (или пометить как завершённый)
2. Сохранить ключевые выводы в MEMORY.md или отдельные файлы памяти
